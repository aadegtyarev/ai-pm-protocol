## Scenarios

1. **The protocol declares a symmetric dual-harness architecture.** A reader of `doc/architecture.md` sees a recorded decision: the protocol is one repository serving two harnesses (Claude Code + OpenCode) via a **harness-neutral core** plus a **thin per-harness adapter**, kept **dual-native** (both adapters committed as plain files, no build step) and guarded by an **equivalence test**. The two adapters are **symmetric** — neither harness is primary, neither adapter reaches into the other's directory; each harness reads **only its own** adapter (`.claude/` for Claude Code, `.opencode/` for OpenCode), and `.opencode/` is fully self-contained (it never depends on `.claude/` existing). OpenCode ships as a preview until two named upstream gaps close.

2. **The harness-neutral core is explicitly carved, and the shared body is kept in one place.** The protocol's harness-agnostic surface — `WORKFLOW.md`, `workflow/*.md`, the **bodies** of every `pm-*` agent and command, `doc/_templates/*` — is confirmed and documented as shared-across-harnesses. The harness-specific shell — instruction-entry mechanism, agent/command frontmatter shape, enforcement mechanism, tool-vocabulary, install wiring — is enumerated per harness in a **harness-adapter contract** the architecture document owns. A harness-neutral vocabulary layer lets the core prose name capabilities ("the structured-question tool", "the change-intent route reminder", "the deny-list enforcement") without binding to one harness's primitive names. **Shared-body sync mechanism — single source of truth, no duplicated bodies, ever.** Every shared body exists in **exactly one** physical place; an adapter never carries its own copy. The form depends only on whether frontmatter diverges: a file that is **byte-identical** on both sides (frontmatter and body alike — e.g. a skill whose frontmatter uses only the fields both harnesses recognize) is a **symlink to a single source**, reusing the protocol's already-canonical symlink-delivery mechanism (the recorded "Settings.json / agents / commands delivered via symlink" decisions); a file whose **frontmatter must differ** between harnesses (agents, commands) is a **thin wrapper** — harness-specific frontmatter plus a body that **points at the single neutral body and instructs the agent to read-and-follow it** (the protocol's existing Read-on-demand pattern, the same one `workflow/*.md` topic files already use). On the OpenCode side the wrapper can be **hardened** beyond a runtime pointer: an OpenCode agent's prompt can be sourced from a file at the config level (`prompt: {file: …}` in `opencode.json`), a hard load with no runtime hop and still single-source — Spike A scope. The guard the equivalence test now enforces is **single-source integrity**: no adapter agent/command file may carry an inline neutral body — it must be a symlink or a read-and-follow pointer to the one source. (Body **duplication + a drift-check is rejected** — see Key design decisions: a self-maintained protocol cannot safely keep two copies in sync, and duplicating would forfeit the one-repo benefit.)

3. **The Claude Code path is unchanged.** A maintainer running the existing dogfood pipeline on Claude Code sees byte-identical `.claude/` adapter behaviour: same agents, same commands, same `settings.json` hooks, `tests/hooks.sh` green, `@`-import of `WORKFLOW.md` intact. Claude Code remains the self-host harness (PM decision 2026-06-07). The symmetry/no-cross-read decision **removes nothing** the Claude path uses — it only forbids the OpenCode adapter from depending on `.claude/`.

4. **A downstream project on OpenCode can install the protocol as a preview, with supported harnesses declared explicitly.** The repository carries a committed `.opencode/` adapter — agents, commands, an enforcement plugin, an `opencode.json` with an `instructions` array, and an `AGENTS.md` entry — translated from the same neutral core. Which harnesses the protocol supports is stated **declaratively** in the `README` and `AGENTS.md` headers (and folded into the preview label) — a reader learns the supported set from an explicit statement, **not** by inferring it from which adapter folder happens to be present. A prominent **preview label** states OpenCode is not yet enforcement-certified and names the two upstream gaps it waits on.

5. **Installation stays one flow across both harnesses, harness auto-detected.** The install instructions (and any install helper) add the submodule identically, then **detect** which harness the project uses (presence of `.claude` vs `.opencode`, or a flag) and wire the matching adapter — the submodule step and the mental model are the same as today's Claude Code install (PM decision 2026-06-07: auto-detect). The existing Claude path wires its adapter via symlinks into the submodule (agents/commands/settings); the OpenCode path wires its own adapter the analogous way (submodule + `instructions` config entry + the adapter files made discoverable under `.opencode/` — exact link/copy step spike-gated, see scenario 7).

6. **Each of the four OpenCode gaps has a recorded adapter strategy.** The design names, for each gap, exactly how the OpenCode adapter behaves and how it differs from Claude Code: the missing per-prompt route-reminder hook, the missing built-in review/research engines, the missing runtime model override (cross-model — handled via static model pins in OpenCode agent frontmatter), and the unsettled subagent enforcement containment.

7. **The two unsettled dependencies are de-risked by gated spikes.** Two throwaway spikes are scheduled (not assumed): **Spike A** verifies submodule-path sourcing for the install design (whether OpenCode's agent/command/plugin loaders can source from a submodule subdir by config alone, or whether a symlink/copy step into `.opencode/` is required); **Spike B** verifies whether plugin `tool.execute.before` actually constrains a `task`-spawned subagent (the disputed `#5894`). Their outcomes flip the affected design points from `doc-cited (unverified)` to `execution-verified` or trigger the documented fallback. **The `@`-inline question is no longer a spike** — it was resolved against canonical docs (see Key design decisions): in-body `@`-import is **not** available for sharing an *agent* body on either harness, so the shared-body mechanism in scenario 2 relies on symlink + read-and-follow pointer, not on `@`-inline.

## Existing behaviors this feature touches

(What must not break — the dogfood self-host runs on this repo under Claude Code.)

- **The Claude Code adapter (`.claude/agents`, `.claude/commands`, `.claude/settings.json`).** No agent, command, or hook changes shape or behaviour. `tests/hooks.sh` stays green at its current count.
- **The `@`-import chain** `CLAUDE.md` → `@WORKFLOW.md` → on-demand `workflow/*.md`. The neutralization in scenario 2 must not change what a Claude Code session loads.
- **The submodule-delivery + symlink install** (the recorded "Settings.json / agents / commands delivered via symlink" decisions and "WORKFLOW.md imported via `@`"). The OpenCode install path is added **alongside**, never replacing the Claude path; the symmetry decision (scenario 1) forbids the new path from depending on the old one's files.
- **The "Agents are plain Markdown files, no build step" architectural decision.** The dual-native choice (symlink / read-and-follow pointer, never a generator) exists precisely to preserve it.
- **The tooling-submodule / dev-artifact inertness invariant** (`template-dev-artifacts-inert`, #251): adapter files for one harness must be inert under the other harness's session, the same way the tooling submodule is inert downstream.

## Stack expectations touched

(From `doc/stack-notes.md` § "OpenCode" — the load-bearing rules the adapter design must respect. Confidence is per-rule below; the two genuinely-unsettled rules stay spike-gated.)

- **Instruction entry — config-listed file array, not in-file import** (`doc-confirmed` 2026-06-07). OpenCode's `opencode.json` `instructions` array accepts relative/glob paths, so a submodule-relative `.ai-pm/tooling/WORKFLOW.md` entry is expressible; but there is **no `@`-import inside an instruction file** ("opencode doesn't automatically parse file references in `AGENTS.md`"). The adapter re-expresses the Claude `@`-import as a config-array entry + the existing harness-agnostic Read-on-demand pattern for topic files. Source: <https://opencode.ai/docs/rules/>, <https://opencode.ai/docs/config/>
- **Subagent frontmatter shape differs; no `@`-inline in an agent body** (`doc-confirmed` 2026-06-07). `.opencode/agent/*.md` uses `mode`/`model`/`permission` and a `tools` **object map** (`{write: true}`, deprecated in favour of `permission`), not Claude's comma-list; invocation is by name/description via the `task` tool (no `subagent_type` string). The agent **body does not process an `@path` inline** (only a config-level `{file:...}` exists for the `prompt` option, not inside the markdown body) — so an agent body cannot be shared by `@`-import; it is shared by symlink (when byte-identical) or by a read-and-follow pointer. Source: <https://opencode.ai/docs/agents/>
- **Command body supports `@file` inline; agent body does not** (`doc-confirmed` 2026-06-07). An OpenCode **command** body can inline a file: "The file content gets included in the prompt automatically." This is an OpenCode-command-only capability (the Claude analogue is `` !`<cmd>` `` dynamic injection, a *different* syntax) and does **not** generalize to agent bodies on either harness. Recorded so the adapter does not over-apply it. Source: <https://opencode.ai/docs/commands/>
- **Enforcement is JS-throw-to-block in a TS plugin** (`doc-cited (unverified)`), not a JSON `permissionDecision` contract: `tool.execute.before` throws to deny / mutates `output.args` to rewrite. Source: <https://opencode.ai/docs/plugins/>
- **No UserPromptSubmit-equivalent hook** (`doc-cited (unverified)`) — no hook injects context per prompt; message hooks are reactive/post-fact. Source: <https://opencode.ai/docs/plugins/>
- **`question` tool = AskUserQuestion equivalent (PRESENT)** and **`skill` tool = Skill equivalent (PRESENT)** (`doc-cited (unverified)`). The protocol's "surface forks via the structured-question tool" and skill-invocation rules port. **The adapter does NOT rely on OpenCode's `.claude/skills/` cross-read** — per the symmetry decision (scenario 1) the OpenCode adapter ships its own skill files under its own path; cross-read is rejected because it would make `.opencode/` depend on `.claude/`. Source: <https://opencode.ai/docs/tools/>, <https://opencode.ai/docs/skills/>
- **No runtime per-`task` model override** (`doc-cited (unverified)`; PR `#17577` closed-not-merged); subagent model is pinned in agent frontmatter or inherited. The Cross-model-review mechanism must pin model in **agent frontmatter** on OpenCode, not via a runtime override. Source: <https://opencode.ai/docs/agents/>, <https://github.com/anomalyco/opencode/pull/17577>
- **No built-in `code-review` / `deep-research`** (`doc-cited (unverified)`) — must be re-bound to a protocol-shipped OpenCode agent/skill. Source: <https://opencode.ai/docs/agents/>
- **Subagent tool-hook containment unsettled** (`doc-cited (unverified)`; `#5894` closed-disputed, fix PR `#7473` closed-not-merged): a plugin `tool.execute.before` may not constrain every action a `task`-spawned subagent takes. **Spike B.** Source: <https://github.com/anomalyco/opencode/issues/5894>
- **Install — submodule-path sourcing for loaders unconfirmed** (`doc-cited (unverified)`). Config can point `instructions` at a submodule path, but the docs do **not** confirm the agent/command/plugin loaders can source from an arbitrary submodule subdir by config alone — a symlink/copy step into `.opencode/` may be required. **Spike A.** Source: <https://opencode.ai/docs/config/>

## Interaction scenarios

(Not isolated — two adapters share one repository, one submodule-delivery channel, one CI/validation surface.)

- **When a Claude Code dogfood session runs in this repo while the `.opencode/` adapter is present:** the OpenCode files are inert — they do not register as Claude agents, do not alter `@`-import loading, and `tests/hooks.sh` is unaffected (mirrors the `template-dev-artifacts-inert` invariant).
- **When an OpenCode session runs against the same tree:** the `.claude/settings.json` hooks and `@`-import are inert for it; OpenCode reads its own `AGENTS.md` + `instructions` + `.opencode/` adapter, and **only** its own adapter (the symmetry/no-cross-read invariant).
- **When a downstream project runs `git submodule update --remote`:** both adapters arrive together in one update; neither harness's install is left stale by a bump.
- **When the harness-neutral core (a `pm-*` agent/command body, `WORKFLOW.md`, a `workflow/*.md` rule) is edited:** because every shared body is single-source (symlink or read-and-follow pointer), the edit lands once and both harnesses pick it up with no second copy to update; `single-source-integrity` fails only if someone has illicitly inlined a body into an adapter. The guard that keeps "one source of truth" literally true without a build step.
- **When the install helper runs in a project that already has a `.opencode/` or `.claude/` dir / where a symlink target exists:** install detects the harness and does not clobber an existing adapter (failure-path, see Test plan).

## Test plan

- **Existing tests that must pass:** all of `tests/hooks.sh` (current count, unchanged — the Claude adapter is byte-identical).
- **New tests:**
  - `single-source-integrity`: given the `.claude` and `.opencode` adapters, when each `pm-*` agent and command file is inspected, then its body is **only** a symlink to the single neutral source or a read-and-follow pointer at it — **never an inline copy of the neutral body**. This is the single-source guard: it fails loudly the moment any adapter file carries a duplicated body (the maintenance trap a self-maintained protocol must never allow), and confirms both adapters resolve to the *same* one source.
  - `opencode-adapter-inert-under-claude`: given a simulated Claude Code session over this tree, when `tests/hooks.sh`'s boundary/route checks run, then the `.opencode/` files neither register as Claude agents nor change any hook outcome (mirrors the dev-artifact-inert invariant).
  - `opencode-adapter-self-contained`: given the `.opencode/` adapter, when its agent/command/skill files are scanned, then **none** references a `.claude/` path — the symmetry/no-cross-read invariant is mechanically enforced (a positive check that `.opencode/` never reaches into `.claude/`).
  - `preview-label-and-supported-harnesses-present`: given the OpenCode adapter, when its entry surface (`AGENTS.md` / `README` / adapter README) is read, then the preview label, the two named upstream gaps, **and the explicit supported-harness declaration** are present — a positive-presence check so "preview" can never silently read as "certified" and the supported set is never left to folder-inference.
- **Interaction scenario tests (one per Interaction scenario above):**
  - `submodule-bump-delivers-both`: simulate a `submodule update`; assert both adapter trees are present and self-consistent after one bump.
  - `inline-body-trips-integrity`: replace a wrapper's read-and-follow pointer with an inline copy of the neutral body in one adapter; assert `single-source-integrity` fails (guard is live, not vacuous — it catches exactly the duplication the design forbids).
  - `install-no-clobber`: run the install helper where an adapter dir / symlink target already exists; assert it detects and refuses to overwrite, with a clear message.
- **Stack-spec tests (one per stack expectation above, verifying against the cited OpenCode contract, not the adapter's own mapping):**
  - `oc-agent-frontmatter-shape`: each `.opencode/agent/*.md` carries a `description` + a valid `mode` + a `permission`/`tools`-object (not a Claude comma-list). Comment cites <https://opencode.ai/docs/agents/>.
  - `oc-no-at-inline-in-agent-body`: no `.opencode/agent/*.md` body relies on an `@path` inline to pull its shared body (the body is shared via symlink or read-and-follow pointer) — freezes the doc-confirmed "no agent-body `@`-inline" fact. Comment cites <https://opencode.ai/docs/agents/>.
  - `oc-instructions-points-at-submodule-core`: `opencode.json` `instructions` includes the submodule-relative always-on core path(s). Comment cites <https://opencode.ai/docs/config/>.
  - `oc-cross-model-pinned-in-frontmatter`: any OpenCode review/audit agent that must run cross-model carries an explicit frontmatter `model:` (no reliance on a runtime override). Comment cites <https://github.com/anomalyco/opencode/pull/17577>.
  - `oc-enforcement-plugin-throws`: the OpenCode enforcement plugin's `tool.execute.before` throws (or mutates args) for each deny-list case that `tests/hooks.sh` covers on the Claude side. Comment cites <https://opencode.ai/docs/plugins/>.

  > Note: these are **form** checks against the documented contract. Behavioural truth for the two unsettled rules (submodule-path sourcing; subagent hook containment) comes from the spikes, not a static test — the static tests freeze the documented shape, the spikes verify it runs.

## Out of scope

- **Full OpenCode certification / "fully-supported" status** — this slice ships a labeled preview only. Certification is a later plan, gated on the two upstream blockers below.
- **Closing the two upstream blockers ourselves** — runtime per-`task` model override (PR `#17577`, closed-not-merged) and subagent hook containment (`#5894`, disputed). We design around them and wait/track; we do not fork OpenCode to fix them.
- **Dogfooding the protocol on OpenCode** — PM decision 2026-06-07: Claude Code stays the self-host harness; OpenCode parity is validated by a smoke/preview path, not by self-hosting.
- **Generator / build-step single-source** — explicitly rejected (PM decision: dual-native + single-source) to preserve the recorded "no build step" architecture decision.
- **Duplicated shared bodies (the old "duplicate + drift-check" fallback)** — explicitly rejected (PM decision 2026-06-07, pass 2). A self-maintained protocol cannot safely keep two copies in sync, and duplication would forfeit the one-repo benefit. Single source of truth is mandatory; there is no duplication fallback.
- **Aligning the command/skill primitive across both harnesses** — keeping the protocol's `/pm-*` as Claude *commands* (recorded rationale: deliberate developer-operator shortcuts, not the PM interface, not model-auto-invoked) is unchanged here. Whether to converge the primitive on "skill" on both sides (both harnesses have a skill concept) is a **separate future step**, not this plan. The OpenCode adapter may express the `/pm-*` procedures as OpenCode commands (which support `@file` inline) without prejudging that convergence.
- **`.claude/skills/` cross-read as a reuse shortcut** — rejected by the symmetry decision; the OpenCode adapter ships its own files rather than reading Claude's.
- **Live execution of the two spikes if no OpenCode runtime is available in-session** — they are scheduled as gated tasks; if no runtime, the first task records the runtime requirement and the affected design points stay `doc-cited (unverified)` with the documented fallback.
- **Sibling harnesses of the `harness` category** — Cursor, Codex CLI, Aider, Continue, and any other AI-coding harness. This plan focuses on **OpenCode** (PM-named); each other harness is a separate plan with its own adapter contract. The dual-harness architecture is designed to **generalize** to N adapters, but only the OpenCode adapter is built here.

## Key design decisions

- **Four PM forks resolved 2026-06-07, pass 1 (AskUserQuestion):** (1) scope = design + groundwork (OpenCode adapter as preview); (2) dogfood = Claude Code stays self-host, OpenCode = downstream-supported; (3) sync = dual-native + equivalence test (no build step); (4) install = auto-detect harness.
- **Symmetry / no cross-read (PM decision 2026-06-07, pass 2).** The two adapters are symmetric — neither is primary, neither reads the other's directory. `.opencode/` is fully self-contained. The earlier "reuse via OpenCode's `.claude/skills/` cross-read" idea is **rejected**: it would make Claude the first-class harness and OpenCode a guest reaching into a foreign folder, contradicting the universality the PM is buying ("open in Claude → write in Claude; open in OpenCode → write in OpenCode"). Enforced by `opencode-adapter-self-contained`.
- **Shared-body sync mechanism — single source of truth, no duplicated bodies (PM decision 2026-06-07, pass 2).** Every shared body lives in **exactly one** physical place; the form depends only on whether frontmatter diverges:
  - *whole file byte-identical (frontmatter included)* → **symlink to a single source**, reusing the protocol's already-canonical symlink-delivery mechanism (the recorded "settings.json / agents / commands delivered via symlink" decisions — so this is not a new mechanism, it is the existing one).
  - *frontmatter must differ (agents, commands)* → **thin wrapper** (harness-specific frontmatter) whose body is a **read-and-follow pointer** at the single neutral body (the Read-on-demand pattern `workflow/*.md` already uses). This is the PM's primary choice. The OpenCode side can be **hardened** to a hard, no-hop load via config `prompt: {file: …}` (still single-source) — confirmed available for the OpenCode `prompt` option (source: opencode.ai/docs/agents); whether to use it per-file is Spike-A / arch-note scope. The Claude side uses the read-and-follow pointer; the irreducible per-harness frontmatter (a few lines: `name`/`tools` vs `mode`/`permission`/`model`) is **not** duplication of meaningful content — it is the genuinely harness-specific shell, which cannot be shared by construction.
- **Sync mechanism — FORM REOPENED (PM, 2026-06-07, pass 3) — UNDECIDED, pending the `pm-architect` arch note.** The *single-source* and *no-duplication* decisions above are firm; what is reopened is **how** single-source is realized. Two candidates are on the table, both genuinely single-source:
  - **A — runtime single-source (the symlink + read-and-follow pointer above).** No build step; edits to the neutral source take effect immediately (best for the self-host dogfood loop); plain files throughout. Weaknesses: the read-and-follow pointer is "soft" (a system-prompt instruction the agent must obey, not a guaranteed load) and symlinks are fragile on Windows checkouts.
  - **B — build-time single-source (PM idea, pass 3): a metalanguage / neutral source + a CI generator that assembles both full adapters.** Still single-source (you edit the source, never the generated files) and it removes A's weaknesses — the generated adapters are **hard, self-contained files** (no runtime pointer, no symlink). Costs: it **reintroduces a build step** (contradicting the recorded "Agents are plain Markdown, no build step" architecture decision — adopting B means *changing* that decision), adds a generator + CI toolchain to a markdown-prose repo that today has only `tests/hooks.sh`, and imposes a "edit source not generated files + regenerate-before-test" discipline on the **self-maintaining** model (trades duplication-burden for toolchain-burden).
  - **C — build-time, TWO repos: split source / distribution (PM idea, pass 3, second variant).** Reorganize the protocol repo to be developed as plain "source code" (the metalanguage / neutral source + the generator), and publish the **built ready-to-use adapters to a *separate* distribution repo** that downstream projects add as the submodule. CI on the server builds source → pushes artifacts to the dist repo. "Just coding here, build on the server." Difference from B: artifacts live in their own repo, not committed alongside source. **Pros:** the dev tree is clean source only (no generated files mixed in, no "is this generated?" confusion, no diff-noise); the downstream submodule is lean (only built adapters, no source/generator); still no downstream build. **Cons:** **two repos to version, release-coordinate, and CI-credential** (this is a deliberate clean split, *not* the duplication the PM rejected — but it is real operational overhead); and it taxes the **dog-food / self-host loop hardest (see cross-cutting note)**.
  - **Cross-cutting note for B *and* C — the dog-food question is the central trade-off.** This repo's defining identity is that it **develops itself under its own protocol**, and the current plain-Markdown design exists precisely so *the thing you edit IS the thing that runs* (immediate self-host). Any build-time form (B or C) breaks that immediacy: to dog-food a source change you must build it into runnable adapters first. With C this is sharper — the runnable form lives in *another* repo, so self-hosting the source repo means either a local build to produce a runnable `.claude/`, or pulling its own dist repo back as a submodule (a build/pull cycle). The arch note must treat "how does the protocol keep dog-fooding itself" as a first-class criterion, not an afterthought. **Nothing technical blocks B or C** — both are established patterns (commit-built-artifacts; split source/dist repo); the question is whether the build/dist machinery is worth it for a self-hosted markdown-prose protocol, vs A's edit=run simplicity.
  - **Submodule feasibility (verified reasoning 2026-06-07):** for **B**, works in exactly one topology — **CI in the protocol repo generates AND commits the adapters; the submodule ships the pre-built plain files; downstream `git submodule update` gets them with NO downstream build** (build-at-downstream-checkout does NOT work — `git submodule update` runs no build hooks and would force the generator toolchain onto every consumer). For **C**, the dist repo is itself the submodule and already holds built files, so downstream is identically build-free; the build simply happens in the source repo's CI before publishing to dist. Both add a CI `git diff`-clean-after-regenerate (B) / source↔dist-in-sync (C) check — the build-time analogue of `single-source-integrity`.

- **Body duplication rejected outright (PM decision 2026-06-07, pass 2) — the earlier "duplicate + drift-check fallback" is removed.** Rationale (PM): this protocol **maintains itself** — the model edits its own neutral core — so two physical copies of a body turn every change into a silent "did you update the other copy?" hazard the protocol cannot see; and if we accept duplication we forfeit the entire point of one repository ("then it's simpler to make two repos"). Therefore there is **no duplication fallback**: the single-source mechanisms above are the only permitted forms, and the `single-source-integrity` test exists precisely to *prevent* a duplicated body from ever creeping in (not to reconcile two that already exist). The read-and-follow pointer's "soft reliability" (it is a system-prompt instruction the agent must obey on first action) is accepted as the price of build-step-free single-source; if it proves weak in practice the answer is to harden the *pointer* (e.g. OpenCode config-file sourcing), never to duplicate.
- **`@`-inline resolved against canonical docs 2026-06-07 — confirmed NOT available for agent bodies (no longer a spike).** A third candidate (`@`-inline the shared body into the wrapper, no hop, no duplication) was investigated and **ruled out for agents on both harnesses**: Claude Code `@`-import is documented *only* for `CLAUDE.md` ("Imported files are expanded and loaded into context at launch") — agent and command/skill bodies do not process `@`-imports (sources: code.claude.com/docs/en/memory, /sub-agents); OpenCode agent bodies have no `@path` inline either (source: opencode.ai/docs/agents). **Asymmetric exception, recorded so the adapter does not over-generalize:** OpenCode *command* bodies do support `@file` inline (opencode.ai/docs/commands), and Claude commands support `` !`<cmd>` `` shell-output injection — but these are different per-harness syntaxes, so they never yield one shared file and do not change the agent-body conclusion. Net: the shared-body mechanism rests on symlink + read-and-follow pointer (with dup+test fallback), never on `@`-inline.

"Visible and honest, not guessed from layout." Enforced by `preview-label-and-supported-harnesses-present`.
