# AP-27. Hallucinated decision component (Layer 2 cross-doc bounded)

**Что нельзя:**

- Planner создаёт ADR с concrete building block / module / API endpoint / schema entity / state machine / pattern в `## Decision` секции, который не traced к source artifact (spec scenario / plan section / foundational invariant / existing ADR).
- Decision component появляется как «factual statement» без alternative framing — выглядит как «design has components A, B, C» (хотя B — invented), а не как «we considered B, accepted because Y».
- Component name выглядит как «elegant generalization» (e.g. `three-tier-envelope`, `unified-context-manager`, `cross-cutting-abstraction`) и принимается оператором без cross-check на ground truth — потому что **plausibility bias** ↑ overrides skepticism.

**Почему:** Layer 1 (frontmatter `spec_reference:`) ловит **visible** drift (alternatives без source). Live test 2026-05-25 показал второй класс — **invisible** drift: hallucinated **decision component** (vs hallucinated **alternative**). ADR проходит Layer 1 (frontmatter валидный, есть `spec_reference:` на spec'а — потому что spec упомянул concept), но **содержимое** Decision секции вводит component который spec не упоминает. PM (не читает код) не имеет mechanism это поймать post-hoc — drift'нет в schema → code → tests на месяцы.

**Failure mode common pattern:** AI замечает «логичную симметрию» («если есть wrap_1, wrap_2 — должен быть wrap_3 для completeness») или «architectural maturity» pattern и **молча реализует** в Decision section. Plausibility signal (visual symmetric / mature) inverse correlated с actual quality когда invented.

**Mode:** *requires-operator-touch (declared decision)*, не hard block. Decision component allowed, но **требует** trace к source через explicit reference.

**Как поступать вместо:**

1. **ADR template обязательная `### Components` subsection** (`doc/_templates/0000-adr-template.md.tmpl`) — каждый named building block имеет explicit reference на spec / plan / foundational / existing ADR.

2. **Linter enforcement** через `check-spec-discipline.sh adr-decision-component-bounded` (family `cross-doc-bounded`):
   - Если есть `### Components` subsection — каждая entry должна содержать reference indicator (`<file>.md`, `ADR-NNNN`, citation marker).
   - Если subsection отсутствует но Decision body содержит ≥ 2 backtick-named identifier'ов → warn (recommend Components subsection).

3. **Reviewer Step 2.5 (Layer 2)** runs cross-doc check **до** plan_approved'а — primary catch point.

4. **Planner auto-loads foundational docs** (vision / positioning / mvp-scope / threat-model) — cross-check каждого component против shared invariants, не только spec'а.

**Override:** `[source-bounded-override: <reason>]` marker в HEAD commit body downgrade'ит fail → warn. Pattern identical AP-25.

**Hard floor:** override не освобождает от Layer 2 reviewer Step 2.5 для plans с новыми ADRs — operator-touch обязателен.

**Use case examples:**

- *Legitimate component:* ADR-0013 Decision: «Multi-key envelope». Components subsection: `wrap_master` → `vision.md § E2E invariant — server не decrypt`; `wrap_device` → `positioning.md § per-device key rotation`; `wrap_recovery` → `<feature>_spec.md § Scenario: device loss`. Each component traced — OK.
- *Silent drift (что блокируется):* ADR-0014 Decision: «Three-tier envelope». Components: `wrap_master` / `wrap_device` / `wrap_session`. `wrap_session` появился «for symmetric architecture» — нет в spec'е, нет в foundational invariants. Linter `adr-decision-component-bounded` flag'ает: third entry без reference. AP-27 catch.

**Relationship:**

- **AP-25** — generic principle (artifact extends beyond source); AP-27 — specific class для decision components (vs alternatives).
- **AP-1** (timing) + **AP-24** (LOC threshold) — sibling enforcement; AP-27 closes content drift в Decision section.
- **AP-28** — sibling Layer 2 check (inter-ADR contradiction).
- **AP-30** — Failure mode driver: plausibility bias увеличивает risk AP-27.
