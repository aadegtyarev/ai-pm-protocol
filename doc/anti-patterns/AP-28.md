# AP-28. Inter-ADR contradiction в одной PR (Layer 2 cross-doc bounded)

**Что нельзя:**

- В одной PR создаются N ADRs sequentially planner'ом, каждый atomic + internally consistent, но **между собой** contradictory: ADR-A в Alternatives explicitly отвергает pattern P, ADR-B Decision implements pattern P под другим именем.
- ADR-B принимается потому что in-isolation выглядит valid (frontmatter валидный, есть Decision rationale). Только pairwise read'инг обоих ADRs ловит contradiction.
- AI пишет каждый ADR в isolation — нет mechanism'а cross-check'ить против sibling ADRs в этом же PR.

**Почему:** Live test 2026-05-25 показал что planner написал 5 ADRs в одной session, каждый Layer 1 valid. Reviewer-agent caught: ADR-A.Alternatives explicitly rejected pattern P как «violation of invariant I», ADR-B.Decision вводит component реализующий P под другим именем (renamed). Каждый ADR в isolation Layer 1 valid; contradiction появляется только при pairwise read.

**Failure mode common pattern:** Planner spawns N ADRs для архитектурно-нагруженной фичи, нет explicit «check sibling ADRs» step. AI считает каждый ADR atomic problem. PM-оператор (не читает код, читает ADR-сводки в plan summary) не имеет cognitive overhead для cross-doc pairwise comparison N(N-1)/2 pairs.

**Mode:** hard block (linter), conservative thresholds для false-positive avoidance.

**Как поступать вместо:**

1. **Linter enforcement** через `check-spec-discipline.sh inter-adr-contradiction` (family `cross-doc-bounded`):
   - Pairwise check всех ADRs изменённых в текущем PR (`git diff main...HEAD --name-only`).
   - Extract red-line phrases из Alternatives section ADR-A: `rejected`, `violates`, `prohibited`, `отвергнуто`, `нарушает`, `запрещено`.
   - Extract backtick-named terms (≥ 4 chars) из those red-line lines.
   - Grep этих terms в Decision section ADR-B (B ≠ A в этом PR) — substring match case-insensitive.
   - Match → fail.

2. **Conservative threshold:** ≥ 4-char meaningful term overlap. Better miss ambiguous case чем flood false positives — operator-fatigue cost > value за false positive в этом check'е.

3. **Reviewer Step 2.5 (Layer 2)** runs pairwise check **до** plan_approved'а — primary catch point. Также primary catch для semantic contradictions которые term-overlap heuristic пропускает (e.g. rejected pattern under completely different naming).

4. **Planner discipline** — § «Inter-ADR consistency check» в `planner.md`: перед commit'ом нового ADR — pairwise compare с уже existing ADRs в branch.

**Override:** `[source-bounded-override: <reason>]` marker — для legitimate cases где term overlap coincidental (rare).

**Use case examples:**

- *Legitimate distinction:* ADR-A rejects `polling-based-sync` (latency overhead). ADR-B Decision implements `event-driven-sync` (no polling term). No term overlap → no false positive.
- *Silent contradiction (что блокируется):* ADR-A.Alternatives: «`server-side-decrypt` — **violates** E2E invariant». ADR-B.Decision: «cache layer uses `server-side-decrypt` для performance optimization». Term `server-side-decrypt` matches → fail. Planner forced acknowledge contradiction up-front, не months later на code review.

**Relationship:**

- **AP-25** — generic source-bounded; AP-28 — specific cross-ADR coherence.
- **AP-27** — sibling Layer 2 check (component bounded); AP-28 catches different class drift (inter-doc vs intra-doc).
- **AP-1** — companion (reactive ADR timing); ADRs created proactively without plan-level fork increase AP-28 risk.
