# AP-29. ADR scope creep (multi-feature, Layer 2 cross-doc bounded)

**Что нельзя:**

- ADR для feature F-current вводит Decision component, реализующий functionality для F-future (другой topic, не in scope этого spec'а) — потому что «envelope должна покрывать все будущие сценарии для completeness».
- ADR не имеет explicit binding к feature scope — frontmatter без `feature_topic:` field, или с feature_topic'ом который не correlates с спека'ой откуда planner spawned'ил ADR.
- Plan для F-current объективно не «упирается» в F-future fork, но planner добавляет F-future component «потому что увидел паттерн».

**Почему:** AP-1 в protocol'е говорит «ADR пишется реактивно при упорстве plan'а в fork» — но это soft enforcement. Без structural `feature_topic:` field и component-scope cross-check — scope creep проходит invisibly. На live test'е 2026-05-25 ADR Decision вводил component C который actually F-future functionality (не F-current), оператор accept'ил потому что «mature architecture». Cascade drift'нул бы в data модели F-future built'нутые на C-based assumption.

**Failure mode common pattern:** AI видит spec F-current + foundational docs, identifies «logical completeness gap» к F-future, добавляет component для F-future scope как «completing the architecture». Plausibility bias amplifies — symmetric coverage выглядит как «mature design», не как scope creep.

**Mode:** *requires-operator-touch (declared decision)*, не hard block при override. Cross-feature component allowed если operator explicitly approved через fork-justification (создаётся отдельный ADR с feature_topic = F-future).

**Как поступать вместо:**

1. **ADR template обязательный frontmatter field `feature_topic:`** (`doc/_templates/0000-adr-template.md.tmpl`) — explicit binding к feature scope.

2. **Linter enforcement** через `check-spec-discipline.sh adr-feature-scope` (family `cross-doc-bounded`):
   - Frontmatter `feature_topic:` обязательно. Missing → fail.
   - Для каждого backtick-named component в Decision Components subsection: grep substring across `<doc_root>/features/*_spec.md`. Если mentioned **только** в другом topic'е (не в current `<feature_topic>_spec.md`) → flag scope creep.

3. **Reviewer Step 2.5 (Layer 2)** primary catch.

4. **Planner discipline** — `planner.md` § «Scope creep check»: Components должны trace к scope этого feature_topic. F-future functionality → defer / split в отдельный ADR другого feature_topic.

**Override:** `[source-bounded-override: <reason>]` marker — legitimate use case rare (например, foundational ADR который genuinely applies across multiple topics; в этом случае feature_topic should be explicit umbrella name или ADR moved в Stage C foundational).

**Use case examples:**

- *Legitimate scope:* ADR-0013 feature_topic: `multi-key-envelope`. Components все trace к `multi-key-envelope_spec.md`. OK.
- *Silent scope creep (что блокируется):* ADR-0014 feature_topic: `auth-flow`. Components: `session-manager` (auth-flow scope, OK), `recovery-flow-coordinator` (только в `account-recovery_spec.md`, не в `auth-flow_spec.md`). Linter `adr-feature-scope` flag'ает `recovery-flow-coordinator` как cross-feature. Planner forced: либо defer в отдельный ADR с feature_topic `account-recovery`, либо operator-approve override.

**Relationship:**

- **AP-1** — generic reactive principle; AP-29 — specific structural enforcement через `feature_topic:` binding.
- **AP-25** — generic source-bounded; AP-29 — specific cross-feature scope.
- **AP-27** — sibling (component-level); AP-29 — feature-level scope.
